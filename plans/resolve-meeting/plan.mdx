<RichText id="intro">

# Resolve meetings & keep the vault from drifting

**Objective.** Stop meeting notes from becoming dead ends. Today rough items captured in a scrum (sidebar redesign, "tags on query packs", liveshell tweaks) sit inside the meeting note and never reach a feature doc or the manifest, so they are invisible to your Bases and impossible to find later. This plan adds two project-scoped skills: **`resolve-meeting`** (process one meeting into the right notes) and **`vault-audit`** (find drift on demand). Both reuse what the vault already has — the Task-note shape, the manifests, the templates, and `scripts/vault_metadata.py`.

**Done means.** Running `/resolve-meeting <note>` reads a rough meeting note, asks all its clarifying questions at once, shows you a summary of what it understood plus a routing plan, and — after one approval — files every item into the correct feature doc or manifest section, links the meeting both ways, marks superseded instructions, and fixes frontmatter. `/vault-audit` produces a findings report (loose ends, broken links, metadata violations) without editing anything.

A few conventions are still open — see **Open Questions** at the bottom.

</RichText>

<Callout id="decision" tone="decision">

**Committed approach (settled with you).** Two separate skills, not one. New feature docs **reuse the existing Task note shape** (`## Overview → ## Milestones → ## Tasks`) — no new template. The resolve flow is **summarize → batch all questions → you approve once → execute**, never per-item nagging and never silent auto-edits. The existing-drift cleanup is **deferred**: this plan ships the skills and hands you the audit findings; you choose when to clean up.

</Callout>

<RichText id="route-h">

### Routing logic (phase 4)

Each extracted item is classified once and sent to exactly one home. This is the rule set the skill encodes — it is also what makes items findable afterward, because every home is either a linked feature doc or a manifest section. Templates guide each note but are not a cage: the skill adds or renames sections when an item does not fit the default layout.

</RichText>

<Callout id="reuse" tone="info">

**Reuses, does not reinvent.** Feature-doc creation calls the same logic as the `make-note` skill (timestamped filename, metadata-contract frontmatter). Frontmatter validation shells out to the existing `python3 scripts/vault_metadata.py audit` rather than re-checking fields in the skill. Templates come from `Templates/` (Task, Meeting, Bug). The skill is routing + linking glue, not new machinery.

</Callout>

  caption="Planned shape — the description drives auto-triggering; the body is the procedure."

name: resolve-meeting
description: Process a rough meeting note into the vault. Trigger on "resolve this meeting", "process my scrum notes", "sort out this meeting". Extracts items, batches clarifying questions, summarizes understanding, then on approval routes each item to the right feature doc or manifest, links the meeting both ways, marks superseded instructions, and fixes metadata.
---

# resolve-meeting

## 1. Intake
Read the target note + frontmatter. Grep recent same-org meetings and
feature docs (by organization + wikilinks) for context.

## 2. Understand
Extract every action item. Classify each: feature | task | decision | bug | question.

## 3. Clarify  (STOP — read-only until approved)
Present: (a) a summary of understanding, (b) ALL clarifying questions at once,
(c) the routing plan (what goes where). Wait for one approval.

## 4. Execute
- feature  -> find existing feature doc; update section + add related-tasks link;
             else create via make-note rules in Task shape, set status.
- task     -> manifest, correct section, link related task note.
- decision -> feature doc Decisions; mark superseded instruction with a callout + link.
- set related-tasks on the meeting note (both directions).
- run: python3 scripts/vault_metadata.py audit  -> fix flagged frontmatter.

## Adapt the note, do not force-fit
Templates are a baseline, not a cage. If an item does not fit the template
sections, add or rename sections in the task/feature note to match the need.
Never drop info just because there is no matching section.

## 5. Confirm
Report created/updated notes (links), manifest changes, and items handed back.`}
/>

### Scope & non-goals

</RichText>

<Callout id="nongoals" tone="warning">

**In scope:** the two SKILL.md files, the routing/linking procedure, reuse of `make-note` + `vault_metadata.py` + templates. **Out of scope (deferred):** the one-time cleanup of existing drift, any change to the Bases/Categories views, a new "Feature" template (we reuse the Task shape), and the supersession-marker convention (Open Question below). **Not touched:** no bulk edits to existing notes during this build — `resolve-meeting` only writes when you run it on a specific note.

</Callout>
